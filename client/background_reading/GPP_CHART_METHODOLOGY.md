# GPP SharingOptOut Chart Methodology

**Describes:** `gpp_schema_opt_out_chart.py` vs. `monthly_data_analysis.py`
**Scope:** GPP SharingOptOut (and related fields) chart generation
**Validation dataset:** August 2025 California crawl

---

## Overview

Two scripts exist for charting GPP opt-out field distributions. The old approach (`monthly_data_analysis.py`) reads raw decoded GPP columns and produces before/after GPC bars aggregated across all GPP sections. The new approach (`gpp_schema_opt_out_chart.py`) reads a pre-computed `compliance_classification` JSON column and produces one bar per GPP section/state. The new approach is the authoritative method going forward.

---

## Old Approach: `monthly_data_analysis.py`

### Data Source

Reads `decoded_gpp_before_gpc` and `decoded_gpp_after_gpc` columns, which are Python-dict-like strings parsed with `ast.literal_eval`. GPP field values are floats:

- `0.0` — Not Applicable
- `1.0` — Opted Out
- `2.0` — Did Not Opt Out

### Site Filter (`subject_or_likely_with_privacy_string`)

Implemented in `processing_analysis_data.py` as `ccpa_subjection_privacy_string_breakdown()`. A site is included if it has:

1. At least one third-party request (`third_party_count > 0`)
2. At least one privacy string detected (any of: USPS before/after, OptanonConsent before/after, GPP before/after, or Well-known with GPC present)

Both `know_subject` and `may_be_subject` subgroups are included. The selling/sharing ad network list does not determine inclusion — it only determines the sub-label.

### Chart Output

Two horizontal stacked bars:

- **Before GPC** — distribution of the field value across all qualifying sites before GPC was sent
- **After GPC** — distribution after GPC was sent

GPP sections (usnatv1, uscav1, etc.) are aggregated per unique site, with `opted_out` taking priority if any section reports opt-out. Three categories are shown: Not Applicable (pink), Did Not Opt Out (blue), Opted Out (green).

Sites where the GPP string disappeared or became unparseable after GPC are silently excluded — no "Invalid/Missing" category exists.

### August 2025 California Results (verified)

| Category | Before GPC | After GPC |
|---|---|---|
| Not Applicable | 52 | 46 |
| Did Not Opt Out | 943 | 682 |
| Opted Out | 28 | 327 |
| **Total** | **1,023** | **1,055** |

The `n` shown in the chart title corresponds to the After GPC total (1,055).

## New Approach: `gpp_schema_opt_out_chart.py`

### Data Source

Reads the `compliance_classification` JSON column, generated by `generate_compliance_column.py`. Each row contains a JSON object with per-family classifications. For GPP, this is a list of `{state, field, status}` entries — one per (GPP section × field) combination.

### Schema Statuses

| Status | Meaning |
|---|---|
| `opted_out` | Field was set to opted-out after GPC (after value = 1.0) |
| `did_not_opt_out` | Field was not opted-out after GPC (after value = 2.0, or was 2.0 before and became 0.0) |
| `not_applicable` | Field was 0.0 both before and after GPC (legitimately N/A) |
| `invalid_missing` | GPP string was present/valid before GPC but disappeared or became unparseable after |

### Classification Rules (`gpp_field_status()` in `generate_compliance_column.py`)

| Condition | Status |
|---|---|
| after = 1.0 | `opted_out` |
| after = 2.0 | `did_not_opt_out` |
| after = 0.0 AND before was 1.0 or 2.0 | `invalid_missing` (section disappeared) |
| after = 0.0 AND before was 0.0 or absent | `not_applicable` |
| after = None/absent | `invalid_missing` |

### Site Filter

No filter is applied by default — all rows with a `compliance_classification` column are used. An `--ad-network-filter` flag exists to optionally apply a selling/sharing ad network filter.

### Chart Output

One horizontal stacked bar per GPP section/state (US National, California, Colorado, etc.), showing all four schema statuses. This replaces the Before/After framing because the schema classification already encodes the before→after comparison into a single status per entry.

### August 2025 California Results (schema, unfiltered)

| Section | N/A | Did Not Opt Out | Opted Out | Invalid/Missing | Total |
|---|---|---|---|---|---|
| US National (usnatv1) | 22 | 564 | 254 | 96 | 936 |
| California (uscav1) | 25 | 120 | 82 | 12 | 239 |

## Discrepancy Investigation

We systematically compared the new schema chart against the old After GPC numbers (46 N/A, 682 Did Not Opt Out, 327 Opted Out) to account for all differences.

### Step 1 — Applying the Matching Filter

When the `subject_or_likely_with_privacy_string` filter was applied to the new approach using raw decoded GPP values directly (not the schema), results were:

| Category | Result | Old After GPC | Match? |
|---|---|---|---|
| Opted Out | 327 | 327 | Exact |
| Did Not Opt Out | 684 | 682 | Off by 2 |
| Not Applicable | 46 | 46 | Exact |

Initial hypothesis was that the 2-site gap was due to duplicate domain entries in the local CSV. This turned out to be incorrect — see Step 4 (Resolution) below for the actual cause.

### Step 2 — Schema vs. Raw Value Consistency

A mismatch check comparing schema classification against raw decoded GPP values site-by-site found **zero mismatches**. The schema classification is 100% consistent with the raw decoded values.

### Step 3 — Filtered Schema Chart vs. Old Chart

| Category | Old After GPC | Schema (subject-filtered) | Difference |
|---|---|---|---|
| Not Applicable | 46 | 46 | 0 |
| Did Not Opt Out | 682 | 682 | 0 |
| Opted Out | 327 | 322 | −5 |
| Invalid/Missing | — | 107 | new category |

The 5-site opted-out gap (schema gives 322, old gives 327) arises because the old chart silently dropped `invalid_missing` sites — sites where the GPP string disappeared after GPC. Some of those sites had SharingOptOut=1 in the raw before value and were counted differently by the old aggregation logic. The row-vs-site_id deduplication difference also contributes.

### Step 4 — Root Cause of the 684 vs. 682 Gap (RESOLVED)

**The gap is fully explained by an error-column filter in `monthly_data_analysis.py` that is absent from the local CSV.**

#### The error filter in the old script

`monthly_data_analysis.py` line 129 filters the merged dataframe before analysis:

```python
added = full_merged_df[
    (full_merged_df['status'] != 'not added') &
    ((pd.isnull(full_merged_df['error']) == True) | (full_merged_df['error'] == 'singleTimeoutError'))
]
```

Only sites with a null error value or a `singleTimeoutError` are retained. Sites with any other error type are excluded from every downstream count. After this filter, `added` = 9,893 rows (not the full 11,708).

#### Error distribution in the server CSV

The server CSV (downloaded directly from Google Sheets) has the following error values for California sites:

| Error value | Site count |
|---|---|
| `null` → NaN (after `replace('null', np.nan)`) | 1,252 |
| `ReferenceError` | 550 |
| `doubleTimeoutError` | 7 |
| `WebDriverError` | 4 |
| `UnexpectedAlertOpenError` | 1 |
| `InsecureCertificateError` | 1 |

#### The two excluded sites

Exactly two sites that would otherwise qualify as `SharingOptOut = did_not_opt_out` are removed by the error filter:

| Site | Error | Effect |
|---|---|---|
| glidemagazine.com | `doubleTimeoutError` | Excluded → count drops by 1 |
| bugsnag.com | `ReferenceError` | Excluded → count drops by 1 |

This fully accounts for the 684 → 682 difference.

#### Local CSV vs. server CSV

The local CSV (at `client/public/CA/`) stores all error values as the literal string `'null'`. Because `replace('null', np.nan)` converts that string to NaN, every site appears error-free in the local CSV — including glidemagazine.com and bugsnag.com — producing a count of 684 instead of 682.

The server CSV is therefore the authoritative source for legacy-mode chart generation. The `--legacy` flag in `gpp_schema_opt_out_chart.py` applies this error filter correctly when using the server CSV.

#### Full replication confirmation

Once the error filter was applied using the server CSV, all numbers matched the old chart exactly:

| Category | Old chart | Full replication | Match |
|---|---|---|---|
| Not Applicable | 46 | 46 | exact |
| Did Not Opt Out | 682 | 682 | exact |
| Opted Out | 327 | 327 | exact |
| Total (n) | 1,055 | 1,055 | exact |

### Step 5 — Why the Unfiltered New Chart Shows Higher Numbers

The unfiltered new chart (US=936, CA=239, total 1,175 section-pairs) exceeds the old n=1,055 for three reasons:

1. No `subject_or_likely_with_privacy_string` filter → more sites included
2. Counting unit is (site × section) pairs, not unique sites — sites with both usnatv1 and uscav1 are counted twice
3. Includes the `invalid_missing` category (108 sites) that was silently excluded from the old chart

## Key Differences Summary

| Aspect | Old (`monthly_data_analysis.py`) | New (`gpp_schema_opt_out_chart.py`) |
|---|---|---|
| Data source | `decoded_gpp_before_gpc` / `decoded_gpp_after_gpc` | `compliance_classification` JSON column |
| Chart framing | Before GPC / After GPC (two bars) | Per GPP section/state (one bar each) |
| Site filter | `subject_or_likely_with_privacy_string` (any PS + any 3rd party) | None by default; optional flags |
| Counting unit | Unique sites (aggregated across sections) | (site × section) pairs |
| Invalid/Missing | Silently excluded | Explicit category (amber) |
| Standalone | No (requires Colab + Google Drive) | Yes (pure Python, standard CSV input) |
| Multi-state | No (one state at a time) | Yes (multiple CSVs side-by-side) |

---

## Why the New Approach Is Authoritative

- Zero mismatches between schema classification and raw decoded GPP values confirm correctness of `generate_compliance_column.py`.
- N/A and Did Not Opt Out match exactly when the same filter and counting method is applied.
- The `invalid_missing` category makes explicit what the old chart silently omitted, improving transparency about data quality.
- Per-section breakdown is more informative than the cross-section aggregated before/after view.
- The script is standalone (no Colab dependency) and supports multi-state analysis from a single invocation.
